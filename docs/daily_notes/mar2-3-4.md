# Retry Sisteminin Evrimi

Başlangıçta retry mantığı şuydu:

```
FAILED → next_retry_at → QUEUED → retry
```

Ama eksik olan şey:

* Hata türü ayırt edilmiyordu
* Tüm hatalar retry ediliyordu
* Sonsuz retry riski vardı
* Operasyonel görünürlük yoktu

Bu boşluklar kapatıldı.

# Smart Retry Policy Katmanı

Yeni bir kavram eklendi:

```
Exception Classification
```

Artık sistem hata türüne göre karar veriyor.

### Hata Türleri

| Tür                                  | Davranış                  |
| ------------------------------------ | ------------------------- |
| Permanent (ValueError, validation)   | Direkt EXTINCT               |
| Transient (ConnectionError, Timeout) | Retry                     |
| Unknown                              | Retry + max_retry kontrol |

Bu şu kazanımı sağladı:

> Sistem artık kör değil. Hata tipine göre davranıyor.
